§0Mental model

§0.1Vocabulary: async ≠ parallel ≠ non-blocking

These words get used interchangeably; they name orthogonal properties. Every technology in this sheet is a different combination of them — misdiagnosing which property you actually need is how teams end up on WebFlux for a CRUD service.

TermA property of…MeaningLitmus example
Blocking / non-blockinghow the wait is implementedDoes an OS thread sit parked inside the call until the result arrives — or is the wait registered with the OS (epoll/NIO) and the thread freed to do other work?socket.read() on a platform thread vs an NIO selector loop
Synchronous / asynchronousthe call contractSync: the result is in hand when the call returns. Async: the call returns immediately and the result is delivered later — via callback, future, or resumption.Quote quote(id) vs CompletableFuture<Quote> quote(id)
Concurrentprogram structureMultiple tasks in progress over overlapping time windows. Interleaving on a single thread counts — no second core required.10k coroutines multiplexed on one dispatcher thread
Parallelhardware executionMultiple tasks executing at the same instant on different cores. Parallelism is a subset of concurrency; it buys throughput for CPU-bound work, nothing for waits.list.parallelStream().map(…)
the trapsAsync ≠ non-blocking: supplyAsync wrapping JDBC is asynchronous to the caller while a pool thread blocks all the same. Non-blocking ≠ async: a suspend fun or a blocking call on a virtual thread gives a synchronous contract over a freed thread. Concurrent ≠ parallel: 100k coroutines can interleave on one thread, while parallelStream is parallel yet fully synchronous — the caller waits.

§0.2Who parks whom

All three models solve the same problem — don't hold an OS thread hostage during I/O — at different layers. Coroutines unmount at suspend points (compiler-generated state machines, CPS transform). Virtual threads unmount at blocking JDK calls (JVM-managed continuations). Platform threads never unmount; they just block.

§1Suspend functions

suspend is a compiler-level colored function: the compiler rewrites it into a state machine taking a hidden Continuation<T> parameter (CPS). Sequential-looking code, no callbacks, no futures in signatures. The two Java translations differ radically in ergonomics.

KOTLIN · suspend

suspend fun quote(id: String): Quote =
    withContext(Dispatchers.IO) { repo.find(id) }  // wrap blocking JDBC

suspend fun enrich(id: String): Enriched {
    val q = quote(id)            // sequential, thread released while waiting
    val r = rating(q.issuer)     // plain call syntax, real stack traces*
    return Enriched(q, r)
}
Signature is honest: suspend ⇒ may park. Callable only from a coroutine or another suspend fn — the compiler enforces the color.
≈

JAVA · with virtual threads (JDK 21+)

// Just write blocking code. The VT unmounts during I/O.
Quote quote(String id) { return repo.find(id); }

Enriched enrich(String id) {
    var q = quote(id);           // "blocks" — but only a VT: cheap
    var r = rating(q.issuer());
    return new Enriched(q, r);
}
No function coloring at all — every method can park. Cost: nothing in the signature tells you it does I/O.

JAVA · without VT (CompletableFuture)

CompletableFuture<Quote> quote(String id) {
    return CompletableFuture.supplyAsync(() -> repo.find(id), ioPool);
}
CompletableFuture<Enriched> enrich(String id) {
    return quote(id)
        .thenCompose(q -> rating(q.issuer())
        .thenApply(r -> new Enriched(q, r)));  // nesting to keep q in scope
}
Monadic coloring: futures infect every signature; intermediate values force nesting or ugly tuples; stack traces are executor frames.
nuance*Coroutine stack traces are synthetic across suspension points; enable -Dkotlinx.coroutines.debug (or the IntelliJ debugger's coroutine agent) for creation stack traces. Virtual threads keep true stack traces — a real observability advantage.

§1.1Under the hood: the CPS transform

CPS = continuation-passing style: instead of returning a value, a function receives a callback (a Continuation) representing everything that happens after it, and invokes it with the result. The Kotlin compiler applies this rewrite mechanically to every suspend fn — you write sequential code, it emits the callback machinery.

KOTLIN · what you write

suspend fun enrich(id: String): Enriched {
    val q = quote(id)         // suspension point 0
    val r = rating(q.issuer)  // suspension point 1
    return Enriched(q, r)
}
⇒

WHAT THE COMPILER EMITS (conceptually)

// hidden param added; return type widened to Object
Object enrich(String id, Continuation<Enriched> cont) {
    // one state-machine object per call; label = resume point,
    // fields spill locals that must survive suspension
    switch (sm.label) {
      case 0:
        sm.label = 1;
        Object q = quote(id, sm);   // sm IS the continuation
        if (q == COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED;
        // completed synchronously → fall through, no scheduling
      case 1:
        sm.q = q; sm.label = 2;
        Object r = rating(q.issuer, sm);
        if (r == COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED;
      case 2:
        return new Enriched(sm.q, r);
    }
}
// later, when the async result arrives:
// dispatcher calls sm.resumeWith(result) → re-enters at label
MechanismConsequence
Hidden Continuation parameter on every suspend fnWhy suspend fns can only be called from suspend contexts — callers must have a continuation to pass. This is the function coloring.
State-machine object holds label + spilled localsThe coroutine's "stack frame" lives on the heap — a suspended coroutine costs a small object, not a ~1 MB thread stack.
COROUTINE_SUSPENDED sentinel unwinds the call chainThe thread is freed immediately; resumption re-enters the switch at the saved label. If the callee finishes synchronously, execution falls through with zero scheduling overhead.
Rewrite happens at compile time (Kotlin compiler)Only suspend points can unmount. Loom does the equivalent stack capture at runtime in the JVM — any blocking call unmounts, no signature change. Same idea, different layer (see §0). Also why coroutine stack traces are synthetic: the real stack unwound; what remains is a chain of continuation objects.

§2Builders & scopes

You can't call a suspend fn from nowhere — a builder creates the coroutine and ties it to a CoroutineScope whose Job owns its lifecycle. Java's counterpart is an ExecutorService — except an executor doesn't propagate cancellation or failure between tasks; that's §3.

runBlocking { }Bridge sync→suspend; blocks caller thread until donefuture.join() / plain main() on a VT (Thread.ofVirtual().start() + join)

springIn Spring, don't hand-roll scopes per request: WebFlux and WebMVC (Framework 6.x) both accept suspend fun controller methods and manage the scope for you. For app-lifecycle scopes, create one CoroutineScope(SupervisorJob() + Dispatchers.Default) bean and cancel it in @PreDestroy — the analog of a Spring-managed ExecutorService with shutdown().

§3Structured concurrency

The core contract: a concurrent task cannot outlive the block that started it, failures propagate to the parent, and cancellation propagates to children. Kotlin has had this since 2018; Java is standardizing the same shape via StructuredTaskScope (JEP 505, 5th preview in JDK 25 — API renamed from the JDK 21 ShutdownOnFailure era to open(Joiner)).

KOTLIN · coroutineScope + async

suspend fun dashboard(id: String): Dashboard = coroutineScope {
    val pos = async { positions(id) }
    val pnl = async { pnl(id) }
    Dashboard(pos.await(), pnl.await())
}   // pnl throws → pos cancelled → exception rethrown here

// race — first success wins, loser cancelled:
suspend fun fastest(): Px = coroutineScope {
    select {
        async { primary() }.onAwait { it }
        async { backup()  }.onAwait { it }
    }.also { coroutineContext.cancelChildren() }
}
≈

JAVA · StructuredTaskScope (JDK 25, --enable-preview)

Dashboard dashboard(String id) throws Exception {
    try (var scope = StructuredTaskScope.open()) {   // forks run on VTs
        var pos = scope.fork(() -> positions(id));
        var pnl = scope.fork(() -> pnl(id));
        scope.join();          // any failure → cancel siblings, throw
        return new Dashboard(pos.get(), pnl.get());
    }
}
// race:
try (var s = StructuredTaskScope.open(
        Joiner.<Px>anySuccessfulResultOrThrow())) {
    s.fork(() -> primary()); s.fork(() -> backup());
    return s.join();          // first success; loser interrupted
}
// deadline for the whole tree:
// StructuredTaskScope.open(joiner, cf -> cf.withTimeout(ofSeconds(2)))
JDK 21–24 preview spelling: new StructuredTaskScope.ShutdownOnFailure() + scope.join().throwIfFailed(). Same semantics, older API.

JAVA · without VT/STS (CompletableFuture)

var pos = CompletableFuture.supplyAsync(() -> positions(id), pool);
var pnl = CompletableFuture.supplyAsync(() -> pnl(id), pool);
try {
    CompletableFuture.allOf(pos, pnl).join();
} catch (CompletionException e) {
    pos.cancel(true); pnl.cancel(true);   // manual, and…
    // …cancel(true) does NOT interrupt a running supplyAsync task!
    throw e;
}
No parent–child relationship, no automatic sibling cancellation, no scoping of lifetimes. This is exactly the gap JEP 505 and coroutines both close.

§5Cancellation & timeouts

Both models are cooperative — nothing dies mid-instruction. Coroutines check at suspension points; Java threads check at interruption-aware calls. The failure modes are symmetric: a tight CPU loop ignores both.

KotlinJava (threads / VT)Watch out

§6Channels — CSP-style handoff

A Channel<T> is a BlockingQueue where put/take suspend instead of block. With virtual threads, the plain BlockingQueue becomes a near-perfect equivalent — blocking a VT is what suspension is to a coroutine.

KOTLIN · Channel pipeline

val ch = Channel<Tick>(capacity = 1024)

scope.launch {                       // producer
    feed.collect { ch.send(it) }     // suspends when full
    ch.close()                       // signals end-of-stream
}
repeat(4) {                          // 4 consumers, work-stealing
    scope.launch { for (t in ch) index(t) }   // loop ends on close
}

// select over multiple channels (no JDK equivalent):
select {
    ticks.onReceive  { handleTick(it) }
    orders.onReceive { handleOrder(it) }
    onTimeout(500.milliseconds) { heartbeat() }
}
≈

JAVA · BlockingQueue + virtual threads

var q = new ArrayBlockingQueue<Tick>(1024);
var POISON = Tick.sentinel();          // no close(): use a poison pill

try (var exec = Executors.newVirtualThreadPerTaskExecutor()) {
    exec.submit(() -> { for (var t : feed) q.put(t); q.put(POISON); });
    for (int i = 0; i < 4; i++)
        exec.submit(() -> {
            for (Tick t; (t = q.take()) != POISON; ) index(t);
            q.put(POISON);              // re-inject for next consumer
        });
}
put() parking a VT is cheap — pre-Loom this pattern needed careful pool sizing; now it's idiomatic. Missing vs Kotlin: close() semantics, select{}, conflation.

JAVA · without VT

// Same BlockingQueue code, but each blocked producer/consumer
// pins a platform thread (~1 MB). Beyond a few hundred parties
// you were pushed toward Reactor:
Sinks.Many<Tick> sink =
    Sinks.many().multicast().onBackpressureBuffer(1024);
sink.tryEmitNext(tick);              // producer (check EmitResult!)
sink.asFlux().subscribe(this::index); // consumer
This is precisely the pressure that created the reactive ecosystem — thread scarcity, not love of operators.

§7Flow ↔ Reactor (and why Streams don't count)

Flow<T> is a cold, suspending, asynchronous stream — same conceptual slot as Flux<T>. Nothing runs until collected/subscribed; each collector gets a fresh run; backpressure is implicit (the collector's suspension slows the emitter). java.util.stream.Stream is synchronous pull, single-use, no time dimension — not an equivalent. java.util.concurrent.Flow (JEP 266) is just the Reactive Streams interfaces — the JDK ships no operators; Reactor/RxJava provide them.

KOTLIN · Flow

fun ticks(sym: String): Flow<Tick> = flow {
    while (currentCoroutineContext().isActive)
        emit(poll(sym))              // suspend-friendly emitter
}
.map { normalize(it) }
.filter { it.px > 0 }
.buffer(256)                          // decouple emitter/collector
.flowOn(Dispatchers.IO)               // context for everything ABOVE
.catch { e -> emit(Tick.stale(sym)) }
.retry(3)

// terminal — suspends until complete:
ticks("AAPL").collect { publish(it) }
≈

REACTOR · Flux

Flux<Tick> ticks(String sym) {
    return Flux.<Tick>generate(s -> s.next(poll(sym)))
        .map(this::normalize)
        .filter(t -> t.px() > 0)
        .onBackpressureBuffer(256)
        .subscribeOn(Schedulers.boundedElastic())
        .onErrorResume(e -> Flux.just(Tick.stale(sym)))
        .retry(3);
}
// terminal — returns immediately, work happens on scheduler:
ticks("AAPL").subscribe(this::publish);
Key ergonomic split: collect suspends the caller (sequential mental model); subscribe detaches (callback mental model). Reactor's explicit request(n) protocol crosses process boundaries (RSocket); Flow's suspension does not.

§9Interop bridges — mix stacks without rewriting

You rarely choose one model wholesale. These adapters make Reactor libraries (WebClient, R2DBC, Kafka Reactor) feel native from coroutines, and vice versa.

KOTLIN ⇢ consuming Java async APIs

// kotlinx-coroutines-reactor:
val user: User = webClient.get().uri("/u/{id}", id)
    .retrieve().bodyToMono<User>()
    .awaitSingle()                    // Mono<T> → suspend

val events: Flow<Event> = flux.asFlow()   // Flux<T> → Flow<T>

// kotlinx-coroutines-jdk8:
val px: Px = completableFuture.await()      // CF<T> → suspend

// blocking Java API (JDBC, Solr client…):
val docs = withContext(Dispatchers.IO) { solr.query(q) }
⇄

JAVA ⇠ consuming Kotlin suspend APIs

// expose suspend fn to Java as Reactor types:
fun enrichMono(id: String): Mono<Enriched> =
    mono { enrich(id) }               // builder from -reactor

fun ticksFlux(): Flux<Tick> = ticks("AAPL").asFlux()

// or as a CompletableFuture (kotlinx-coroutines-jdk8):
fun enrichCf(id: String): CompletableFuture<Enriched> =
    scope.future { enrich(id) }
Dependency map: kotlinx-coroutines-reactor (Mono/Flux ⇄ suspend/Flow, incl. Reactor context propagation), -jdk8 (CompletableFuture), -reactive (raw Reactive Streams Publisher).

§10WebMVC vs WebFlux — the post-Loom decision

Pre-Loom, WebFlux's core sell was escaping the thread-per-request ceiling. Virtual threads erase that argument. What remains for WebFlux is semantics: Reactive-Streams backpressure as a first-class protocol, streaming operator composition, and ecosystems that are reactive all the way down. If you don't need those, WebMVC + VT is the simpler system — real stack traces, working ThreadLocals/MDC, plain JDBC, thread-per-request debugging.

ScenarioVerdictWhy
CRUD / REST over JDBC or JPA (the median service)WebMVC + VTJDBC is blocking; on VTs that's fine. R2DBC buys complexity, not throughput, once threads are free.
High-fan-in API aggregator (call 10 services per request)WebMVC + VT (+ coroutines or STS for the fan-out)Concurrency-per-request is structured-concurrency's job, not the web stack's.
SSE / WebSocket push to moderate subscriber countsWebMVC + VT sufficesMVC supports SSE (SseEmitter, or returning Flow/Flux with Spring 6). One parked VT per connection is cheap.
Market-data style streaming with per-subscriber flow control, windowing, conflation at 100k+ connectionsWebFluxYou need the Reactive-Streams request(n) protocol and operator algebra (window, sample, onBackpressureLatest) end to end.
API gateway / proxyWebFluxSpring Cloud Gateway is built on it; byte-level streaming without buffering whole bodies.
RSocket, reactive Kafka (reactor-kafka), R2DBC estateWebFluxBackpressure must cross the process boundary; suspension can't — it's in-process only.
Team writes Kotlin, wants streaming, hates Reactor's operator soupEither stack + coroutinessuspend controllers & Flow returns work on WebFlux (since 5.2) and WebMVC (since Framework 6). WebFlux + coroutines = reactive runtime, imperative code.
